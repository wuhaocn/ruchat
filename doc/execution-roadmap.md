# Execution Roadmap

这份文档用于固定项目后续推进方式、阶段目标和优先级。

目标不是展开长期愿景，而是把后续工作收敛成一条可持续执行的主线，并且约束每一轮工作都按同一套步骤落地。

## 1. 执行规则

从这份文档开始，后续每一项改动都固定按下面顺序推进：

1. 改造
2. 清理
3. 更新文档

含义如下：

- 改造
  先完成目标能力、结构调整和测试补齐
- 清理
  删除旧逻辑、收敛重复实现、修正命名和状态定义
- 更新文档
  同步 README、doc、脚本说明和联调说明，避免代码与文档漂移

这条规则适用于：

- 功能开发
- 架构调整
- 安全改造
- 控制台优化
- AI 接入

## 2. 总体推进顺序

项目后续按下面顺序推进：

1. 主链路稳定化
2. 安全与部署基线
3. 架构解耦与扩展准备
4. 控制台与审计完善
5. AI 最小接入

这里的关键原则是：

- 不先接 AI
- 不退化成任意 shell 远控
- 先把受控任务模型做稳，再往上叠能力

## 3. 阶段目标

### 3.1 主链路稳定化

目标：

- 固定 `bootstrap -> ws -> task -> ack -> result` 主链路
- 补齐自动化测试
- 明确任务状态机和超时语义
- 让错误定位有日志和事件支撑

这一阶段完成后，项目应达到：

- `cargo check` 稳定通过
- 正常环境下主链路测试可回归
- 关键异常路径有明确行为定义

### 3.2 安全与部署基线

目标：

- 去掉明显 PoC 化的默认安全假设
- 补齐 `https/wss` 与反向代理支持
- 强化管理员会话和 node token 基线

这一阶段优先解决：

- 默认弱口令
- 非随机 session token
- 仅支持 `http/ws` 的限制
- 反代场景下 `ws_url` 生成不稳

### 3.3 架构解耦与扩展准备

目标：

- 把当前单进程内存状态从业务流程里解耦
- 为后续多实例或共享存储预留接口
- 让“创建任务”和“派发任务”职责分离

优先抽象：

- `SessionRegistry`
- `AdminSessionStore`
- `TaskDispatcher`

### 3.4 控制台与审计完善

目标：

- 让人工闭环真正适合日常使用
- 让操作员不依赖服务端日志也能定位大部分问题
- 把任务来源、取消原因、重试原因、结果截断等信息展示完整

### 3.5 AI 最小接入

目标：

- 把 AI 放在控制面之上，而不是让 AI 越过控制面
- AI 只能创建受约束的任务
- 保留完整审计和人工接管能力

最小约束如下：

- AI 只能选定 node
- AI 只能选择该 node 已声明命令
- AI 只能填写参数和 timeout
- AI 不能构造任意 shell

## 4. 当前优先级

当前优先级按 P0 / P1 / P2 划分。

### 4.1 P0

- 跑通 HTTP/WS 集成测试
- 固定任务状态机
- 修正 timeout 语义
- 去除默认弱口令依赖
- 管理员 session 改为强随机 token
- 补全 TLS 与反代支持
- 抽象 `SessionRegistry`
- 抽象 `TaskDispatcher`
- 增强任务详情和控制台核心视图

### 4.2 P1

- 补客户端执行器异常测试
- 标准化服务端与客户端日志
- 完善 cookie 安全属性
- 设计 token 轮换方案
- 抽象 `AdminSessionStore`
- 增强 node 工作台、筛选与运行态概览

### 4.3 P2

- 扩展系统审计事件类型
- 输出单实例 / 多实例 ADR
- 接入 AI 创建任务入口
- 增加 AI 策略层和 AI 审计字段

## 5. 统一工作模板

后续每个 issue 或每轮开发都应按下面模板描述。

### 标题

`[模块] 目标`

### 改造

- 改哪些文件
- 实现什么能力
- 补哪些测试

### 清理

- 删除哪些旧逻辑
- 收敛哪些重复实现
- 修正哪些命名、状态或配置

### 更新文档

- 更新哪些文档
- 增加哪些示例
- 删除哪些失效说明

### 验收标准

- 功能验收
- 测试验收
- 文档验收

## 6. 推荐的下一轮执行范围

如果按当前阶段继续推进，建议先只做下面一轮：

1. 跑通 HTTP/WS 集成测试
2. 固定任务状态机
3. 修正 timeout 语义

原因很简单：

- 这三项直接决定当前主链路是否稳定
- 后续安全改造和 AI 接入都依赖这条链路
- 现在先做控制面加固，收益最高
